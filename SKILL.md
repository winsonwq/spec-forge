# SPEC-FORGE

把模糊想法锻造成结构化、可测试、可执行的规格文档，并支持持续迭代改进。

## Intake Interview（入口）

当 skill 被触发时，首先进行 intake interview：

1. 这个 spec 的主题是什么？一句话描述
2. 目标用户/受众是谁？
3. 是技术方案文档还是需求规格文档？
4. 有没有特定的技术栈或约束？
5. 是否涉及以下维度：
   - 算法逻辑/复杂计算？
   - 复杂 UI/交互？
   - 业务流程/状态机？
6. 个人使用还是团队协作？
7. 简要说明（将用于文件名）：{简短说明}

## Subagents（6个）

| Agent | 职责 | 触发条件 |
|-------|------|----------|
| `requirement-drafter` | 把模糊想法写成结构化 SPEC | 每次都需要 |
| `logic-verifier` | 用代码验证算法/业务逻辑 | 涉及算法/复杂逻辑时 |
| `ui-verifier` | 用组件树验证 UI/交互 | 涉及复杂 UI 时 |
| `flow-verifier` | 用流程图验证业务流程 | 涉及业务流程时 |
| `quality-checker` | 检查完整性/可测试性/逻辑一致性 | 每次都需要 |
| `task-breaker` | 把 spec 拆成 2-3 级任务 | 需要任务拆解时 |

## Reactive Loop（调度逻辑）

每完成一个 agent → 检查产出 → 决定下一步：

```
requirement-drafter 完成后
  → quality-checker 检查
    → FAIL → 根据问题触发对应 verifier（logic/ui/flow）
    → PASS → 记录版本，进入下一阶段

任何 verifier 完成后
  → quality-checker 复查
    → 通过 → requirement-drafter 更新 spec
    → 继续 → 循环直到通过

所有检查通过后
  → task-breaker 进行任务拆解
  → 产出最终版本
```

## 冲突解决

当不同 agent 的结论矛盾时：
- 逻辑优先：logic-verifier 的结论优先（代码能跑过是硬证据）
- 用户仲裁：无法判断时，抛给用户决定
- 文档记录：冲突点和解决方案记录到 changelog

## 版本管理

- 格式：X.Y.Z（语义化版本）
- 补丁（Z+1）：修复错别字、格式调整等小改动
- 次版本（Y+1）：新增内容、修改描述
- 主版本（X+1）：大幅重构、新增章节

初始版本：v1.0.0

## 产出格式

文件名：`{简短说明}-v{版本}-spec.md`

示例：`user-authentication-v1.0.0-spec.md`

产出为单个 Markdown 文件，包含：
- 元信息（版本、日期、状态）
- 各章节（背景/目标/范围/方案/任务/测试/成功标准）
- changelog（作为末尾章节，仅大版本结束时更新）

## SPEC 基本章节要求

每个 SPEC 必须包含以下章节：

| 章节 | 要求 |
|------|------|
| 背景 | 清晰说明为什么要做这个 |
| 目标 | 可量化，最好符合 SMART 原则 |
| 范围 | 明确做什么 / 不做什么 |
| 方案 | 技术方案/实现思路，逻辑自洽 |
| 任务拆解 | 关键任务及依赖关系 |
| 测试验证 | 具体验证方法和标准 |
| 成功标准 | 可衡量的指标 |
