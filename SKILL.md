# SPEC-FORGE

把模糊想法锻造成结构化、可测试、可执行的规格文档，并支持持续迭代改进。

## 核心循环机制

SPEC-FORGE 是一个反应式循环系统，每一轮迭代都会：
1. 产出新的 SPEC 内容
2. 更新版本号（Z+1 表示小改进）
3. 由 quality-checker 进行质量评估
4. 根据结果决定下一步

循环直到 quality-checker 全部通过。

## Intake Interview（入口）

**intake interview 是整个循环的第一步**，收集的信息决定后续调度哪些 subagent。

通过提问了解：

1. 这个 spec 的主题是什么？一句话描述
2. 目标用户/受众是谁？
3. 是技术方案文档还是需求规格文档？
4. 有没有特定的技术栈或约束？
5. 是否涉及以下维度：
   - 算法逻辑/复杂计算？
   - 复杂 UI/交互？
   - 业务流程/状态机？
6. 个人使用还是团队协作？
7. 简要说明（将用于文件名）：{简短说明}
8. **是否有现有项目代码库？（如有，告知项目路径）**

根据回答，判断需要触发哪些 subagent。

## Subagents（8个）

| Agent | 职责 | 触发条件 |
|-------|------|----------|
| `project-analyzer` | 分析现有项目技术架构（语言/框架/结构/依赖） | 用户有现有项目时 |
| `requirement-drafter` | 把模糊想法写成结构化 SPEC | 每次都需要 |
| `logic-verifier` | 用代码验证算法/业务逻辑 | 涉及算法/复杂逻辑时 |
| `ui-verifier` | 用组件树验证 UI/交互 | 涉及复杂 UI 时 |
| `flow-verifier` | 用流程图验证业务流程 | 涉及业务流程时 |
| `test-designer` | 设计完整测试用例（功能/边界/异常/安全/性能） | 每次都需要 |
| `quality-checker` | 文档质量检查 + 测试用例交叉验证（覆盖度/冲突检测） | 每次都需要 |
| `task-breaker` | 把 spec 拆成 2-3 级任务（用 Markdown checkbox 标注） | 需要任务拆解时 |

## Subagent 调用约定

调用 subagent 时，**必须传递以下上下文**：

```json
{
  "agent": "logic-verifier",
  "spec_content": "当前 SPEC 完整内容",
  "focus_area": "需要该 agent 处理的具体部分",
  "output_path": "产出保存路径",
  "version": "当前版本号"
}
```

每个 subagent 在执行时：
1. 读取传入的 spec_content
2. 专注于 focus_area 指定的内容
3. 产出结果追加/更新到 SPEC
4. 返回结构化的 JSON 结果

## Reactive Loop（调度逻辑）

**每轮循环都会：产出新内容 → 版本号 +1 → quality-checker 评估 → 决定下一步**

```
┌─────────────────────────────────────────────────────────┐
│  Intake Interview                                        │
│  ↓ 收集需求，判断需要哪些 subagent                        │
└─────────────────────────────────────────────────────────┘
                          ↓
            ┌───────────────────────────────┐
            │  是否有现有项目？               │
            └───────────────────────────────┘
                    ↓ 是           ↓ 否
            ┌───────────────┐         (跳过)
            │ project-      │
            │ analyzer      │
            │ 分析技术架构   │
            └───────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  requirement-drafter                                     │
│  ↓ 生成 SPEC 草稿（基于技术架构约束，如有）              │
│  版本号: v1.0.0 → v1.0.1                               │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  test-designer                                          │
│  ↓ 设计完整测试用例                                      │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  quality-checker                                         │
│  ↓ 文档质量检查（完整性/SMART/一致性/歧义）              │
│  ↓ QA 批判性思考（挖掘潜在问题）                          │
│  ↓ 测试用例交叉验证（覆盖度/冲突检测）                    │
└─────────────────────────────────────────────────────────┘
                          ↓
                    ┌─────┴─────┐
                    ↓           ↓
               FAIL           PASS
                    ↓           ↓
     根据问题类型触发对应 agent：
     - 文档问题 → requirement-drafter
     - 测试用例问题 → test-designer
     - 算法问题 → logic-verifier
     - UI问题 → ui-verifier
     - 流程问题 → flow-verifier
          ↓
     处理完成后 → quality-checker 复查
          ↓
     版本号 +1 (Z+1)
          ↓
     继续循环直到全部通过
                          ↓
┌─────────────────────────────────────────────────────────┐
│  task-breaker                                           │
│  ↓ 生成 2-3 级任务拆解（Markdown checkbox）              │
│  版本号: v1.0.x → v1.1.0（次版本）                     │
└─────────────────────────────────────────────────────────┘
                          ↓
                    最终 SPEC 产出
```

**版本更新时机**：
- 每轮循环结束：Z+1（补丁版本）
- 重大改进（新增章节/大幅修改）：Y+1（次版本）
- 里程碑完成：X+1（主版本）

## 冲突解决

当不同 agent 的结论矛盾时：
- 逻辑优先：logic-verifier 的结论优先（代码能跑过是硬证据）
- 用户仲裁：无法判断时，抛给用户决定
- 文档记录：冲突点和解决方案记录到 changelog

## 版本管理

- 格式：X.Y.Z（语义化版本）
- 补丁（Z+1）：修复错别字、格式调整等小改动
- 次版本（Y+1）：新增内容、修改描述
- 主版本（X+1）：大幅重构、新增章节

初始版本：v1.0.0

## 产出格式

文件名：`{简短说明}-v{版本}-spec.md`

示例：`user-authentication-v1.0.0-spec.md`

产出为单个 Markdown 文件，包含：
- 元信息（版本、日期、状态）
- 各章节（背景/目标/范围/方案/任务/测试/成功标准）
- changelog（作为末尾章节，仅大版本结束时更新）
- **任务追踪动态化说明**（见下方章节要求）

## SPEC 基本章节要求

每个 SPEC 必须包含以下章节：

| 章节 | 要求 |
|------|------|
| 背景 | 清晰说明为什么要做这个 |
| 目标 | 可量化，最好符合 SMART 原则 |
| 范围 | 明确做什么 / 不做什么 |
| 方案 | 技术方案/实现思路，逻辑自洽 |
| 任务拆解 | 关键任务及依赖关系 |
| 测试验证 | 具体验证方法和标准 |
| 成功标准 | 可衡量的指标 |
| 任务追踪 | **动态化**：任务完成时在对应 checkbox 旁标注完成时间和 commit（如 `- [x] **T1** @2026-05-01`），执行过程中持续同步 |

### 任务追踪动态化说明

所有 SPEC 在任务拆解章节包含以下声明：

```markdown
> **任务追踪**：本 SPEC 中的任务状态在执行过程中动态更新。
> 完成标志为 `- [x] **T{n}** @YYYY-MM-DD (commit: <hash>)`，包含完成日期和 commit hash。
> 执行者需在每次 commit 时同步更新对应任务状态。
```

任务格式约定（task-breaker 需严格遵循）：
```markdown
- [ ] **T1**: 任务描述...
- [ ] **T2**: 任务描述...
```

完成时更新为：
```markdown
- [x] **T1** @2026-05-01: 任务描述... (commit: abc1234)
```
